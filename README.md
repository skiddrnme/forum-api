# Forum API — учебное задание по Go

Это практическое задание на реализацию полноценного HTTP API на Go.

В проекте уже есть готовая OpenAPI-спецификация форума и набор end-to-end тестов. Ваша задача — написать серверную реализацию, которая полностью соответствует спецификации и проходит все тесты без их изменения.

Задание намеренно сделано достаточно объёмным. Оно охватывает практически все ключевые аспекты разработки API на Go. Цель задания — научиться уверенно реализовывать HTTP API по контракту.Если после его выполнения написание сервисов на Go перестаёт вызывать сложности и ощущается как системная задача, значит цель достигнута.

> На это задание у вас уйдет много времени, будьте к этому готовы. Подобное задания когда-то и я решал, правда оно было еще больше и требовало всяких оптимизаций в сервисе и БД для выдерживания нагрузочных тестов:)

---

## Суть задания

В репозитории присутствует описание API в формате OpenAPI. Также есть E2E-тесты, которые проверяют как happy-path сценарии, так и негативные кейсы.

Необходимо реализовать сервер, который:

— поднимается и принимает HTTP-запросы
— реализует все методы из спецификации
— корректно валидирует входные данные
— возвращает коды и структуры ответов строго по контракту
— проходит все предоставленные тесты

Как именно вы будете писать сервер — полностью на ваше усмотрение.

---

## Варианты реализации

Разрешены любые адекватные подходы.

Можно написать реализацию полностью с нуля, используя стандартный пакет `net/http`. В этом случае вы самостоятельно реализуете роутинг, middleware и обработку ошибок.

Можно использовать любой популярный HTTP-фреймворк для Go. Например `chi`, `gin`, `echo`, `gorilla/mux` или любой другой, с которым вам комфортно работать.

Можно пойти путём кодогенерации и использовать `oapi-codegen`, сгенерировать интерфейсы или хендлеры из OpenAPI-спеки и реализовать бизнес-логику поверх сгенерированного кода.

Все подходы равноправны. Выбор сделан намеренно, чтобы задание можно было использовать и для обучения, и для сравнения разных стилей разработки.

---

## Что отрабатывается в задании

В процессе реализации вы неизбежно столкнётесь с реальными задачами, которые встречаются в продакшн-API: валидация входных данных, корректная обработка HTTP-статусов, работа с заголовками, идемпотентность запросов, пагинация, поиск, использование `oneOf` / `allOf` в ответах, middleware (`CORS`, `panic-recovery`), а также end-to-end тестирование.

Задание специально построено так, чтобы не сводиться к «написанию CRUD», а дать целостное понимание API-сервиса.

---

## Тестирование

Все проверки выполняются через end-to-end тесты.

Запуск тестов:

```sh
go test -v -count=1 -tags e2e ./...
```

> Тесты менять нельзя. Если тест падает — это означает, что реализация не соответствует контракту, описанному в спецификации.

## Дополнительное задание

В качестве дополнительной практики можно попробовать поднять Swagger UI прямо внутри сервиса, чтобы иметь возможность делать запросы к своему же API через браузер.

Это не является обязательной частью задания, но хорошо помогает в отладке и даёт дополнительное понимание того, как OpenAPI используется на практике.

---

## Рекомендации

Внимательно читайте спецификацию — все требования, которые проверяются в тестах, описаны именно там. Не стоит пытаться угадывать поведение API.

Для более глубокой практики рекомендуется хранить данные не только in-memory, а использовать реальную базу данных (например SQLite или PostgreSQL). Это необязательное требование, но полезное упражнение.

Старайтесь писать код так, как если бы это был реальный сервис, а не учебный пример.
